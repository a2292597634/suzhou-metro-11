## 背景

当前 `js/app.js` 是一个 1300+ 行的单体对象 `BattleMap`，承载了数据管理（加载/保存/导入/导出）、SVG 渲染、DOM 卡片渲染、事件绑定、视口控制、编辑功能等全部职责。代码已出现以下问题：
- 定位功能困难（如"出租率计算在哪"需要全文搜索）
- 无法对单一职责写单元测试
- 多人同时修改不同功能时极易产生合并冲突
- 新增功能（如商铺增删）时难以判断该插入到对象哪个位置

本次重构是纯代码结构重组，**零行为变更**。

## 目标 / 非目标

**目标：**
- 将 `app.js` 拆分为按职责划分的 ES Module，每个模块单一职责
- 消除全局 `app` 和 `BattleMap` 单例，改为模块化导入
- 使每个模块可独立理解、定位、测试
- 保持 HTML/CSS/用户交互行为 100% 不变

**非目标：**
- 不引入任何新依赖（如 React、Vue、构建工具）
- 不改数据模型结构（站点/商铺/全局统计 字段不变）
- 不改后端 API
- 不改 UI 视觉和交互逻辑

## 设计决策

### 1. 使用原生 ES Module（`type="module"`），不引入构建工具
**理由**：当前项目是纯静态站点，引入 Vite/Webpack 会增加复杂度。原生 `import/export` 已足够满足模块化需求，且现代浏览器（Chrome 60+、Edge 79+）已全面支持。后续若迁移到 React 再引入构建工具。

### 2. 模块划分：数据层 / 渲染层 / 交互层 / 工具层
**理由**：按职责垂直拆分，而非按功能水平拆分。数据层管理状态和 I/O，渲染层负责 DOM/SVG 输出，交互层处理用户输入，工具层提供纯函数。

| 模块 | 职责 | 对应原 app.js 行号 |
|------|------|-------------------|
| `data.js` | 站点数据、全局统计、加载/保存、导入导出 | ~23-408 |
| `render.js` | 统计面板、SVG、站点卡片、图例、底部 | ~410-799 |
| `interaction.js` | 事件监听、可编辑元素、模态框 | ~801-1038 |
| `viewport.js` | 缩放、拖动、触控、控制按钮 | ~1040-1276 |
| `utils.js` | 避让算法、出租率计算、HTML 转义 | ~159-176, 546-647 |
| `main.js` | 入口：初始化、模块组装、暴露全局 API | 原 init + 全局 app |

### 3. 保持 `index.html` 中 `onclick` 调用的全局 API
**理由**：HTML 中有大量 `onclick="app.xxx()"` 调用，若全部改为事件委托需要改动 HTML 结构。过渡期内保留全局 `app` 对象作为 facade，由各模块函数代理，后续 React 迁移时再统一替换。

## 风险与权衡

- **风险**：模块化后跨模块调用链变长，调试时需要多文件跳转 → **缓解**：模块数量控制在 6 个以内，职责边界清晰
- **风险**：`import/export` 在旧版浏览器（IE11）不兼容 → **缓解**：本项目面向内部使用，浏览器环境可控
- **权衡**：暂不引入构建工具意味着没有 tree-shaking 和代码压缩 → **接受**：当前代码量不大，后续迁移到 React 时自然会引入
